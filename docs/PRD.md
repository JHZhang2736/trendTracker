## 1. 产品概述

### 1.1 产品背景

当前互联网趋势数据高度碎片化，分散在谷歌趋势、TikTok、微博等多个独立平台。商业决策者需要在多个平台间手动收集、对比、分析数据，耗时且容易遗漏信号。此外，原始趋势数据本身不能直接指导商业行动，需要结合具体业务场景进行解读。

### 1.2 产品定位

TrendTracker 是一款**多平台趋势聚合 + AI商业洞察**工具。将分散在各大平台的热门关键词统一采集、标准化处理，通过 AI 智能管线（过滤噪音 → 重要性评分 → 深度分析）自动筛选出对用户有价值的信息，并结合网络搜索生成深度商业分析报告，辅助用户制定商业计划。

### 1.3 核心设计原则

1. **各平台数据独立展示**：不同平台的热词格式、语义、热度量纲、语言差异极大（微博 = 中文社会热点，Google = 英文搜索词，TikTok = 英文 Hashtag），不强行合并排行，避免语言混杂和量纲误导
2. **平台可插拔**：新增数据源只需实现 `BaseCollector` 接口并注册，无需改动核心逻辑；前端通过中心化配置适配展示
3. **评分平台内归一化**：收敛评分只在同平台内对比，以排名和相对热度为依据
4. **AI 管线化处理**：采集后经过 AI 过滤 → 打分 → 深度分析三级管线，逐步提炼高价值信息

## 4. AI 智能管线设计说明

### 4.1 管线总览

```
采集完成（多平台并行）
    ↓
Stage 1: AI 过滤 + 重要性评分（一次 LLM 调用）
    输入: 30条关键词 + 用户画像
    输出: [{"i": 1, "s": 85, "r": "AI芯片供应链影响科技投资"}, ...]
    未出现的 index → 自动标记 irrelevant, score=0
    ↓
Stage 2: 信号检测（纯算法，无 LLM 调用）
    检测: 排名跃升(≥20位) / 新上榜(Top50首次) / 热度飙升(≥2倍)
    ↓
Stage 3: 深度分析（Top N 条目）
    触发: 自动(采集后取 score 最高 N 条) + 手动(用户点击)
    流程: 搜索引擎获取上下文 → LLM 生成深度报告
    去重: 同关键词 24h 内不重复分析
```

### 4.2 Stage 1 — 过滤 + 打分

将原来的二分类（relevant/irrelevant）升级为带分数的评估：
- LLM 返回精简 JSON：`[{"i": 索引, "s": 分数, "r": "一句话理由"}, ...]`
- 使用短 key (`i/s/r`) 减少 token 消耗，降低截断风险
- 未出现在列表中的条目自动标记为不相关（score=0）
- 多层 fallback：JSON → code fence → regex 配对 → 退化为纯索引模式

### 4.3 Stage 3 — 深度分析

对重要性评分最高的条目进行深度分析：

1. **网络搜索**：用关键词调用搜索引擎（默认 DuckDuckGo），获取 Top 5 结果（标题+摘要+URL）
2. **LLM 深度分析**：将搜索结果 + 热搜元数据喂给 LLM，生成结构化报告
3. **报告结构**：
   - 事件背景（100-200 字）
   - 商业机会（100-200 字）
   - 潜在风险（50-100 字）
   - 建议行动（50-100 字）
   - 情感倾向（positive/negative/neutral）
   - 信息来源（URLs）

## 5. 收敛评分设计说明

### 5.1 设计原则

各平台热度量纲差异极大，不可直接横向比较：

| 平台 | 热度字段 | 典型量级 |
|------|----------|---------|
| 微博 | 热搜指数 | 万级 |
| Google | 搜索量估算 | 百万级 |
| TikTok | 视频播放量 | 十亿级 |

因此评分**只在同平台内部对比**，含义为"这条词在当前平台有多热"。

### 5.2 计算公式

```
热度分(50%) = 该词热度 / 同平台同批次最高热度 × 100
排名分(50%) = max(0, (50 - (rank+1)) / 50 × 100)

原始分 = 热度分 × 0.5 + 排名分 × 0.5
收敛评分 = 原始分 × exp(-age_hours × ln2 / 12)   # 半衰期 12 小时
```

排名是天然跨平台可比的（第 1 名语义相同），热度通过平台内归一化消除量纲差异。

## 6. 平台扩展规范

新增一个数据源的完整步骤：

**后端（2 个文件）**：
1. `backend/app/collectors/{platform}.py` — 实现 `BaseCollector`
2. `backend/app/collectors/{platform}_mock.py` — 实现 Mock 版（用于测试）
3. 在 `backend/app/collectors/__init__.py` 注册

**前端（1 处配置）**：
在 `frontend/lib/platform-config.ts` 的 `PLATFORM_CONFIG` 中添加一条记录（展示名、颜色、图标）

## 7. 非功能性需求

| 类别 | 要求 |
|------|------|
| 性能 | 仪表盘首屏加载 < 3 秒；采集任务并发执行，耗时取最慢平台 |
| 可用性 | 单平台采集失败不影响其他平台和整体服务 |
| 数据保留 | 默认保留 90 天历史趋势数据 |
| 可扩展性 | 新增平台、新增 AI Provider、新增搜索引擎均不修改核心逻辑 |
| 资源占用 | 在普通个人电脑（8GB 内存）上可流畅运行 |
| AI 容错 | LLM 调用失败不阻塞采集流程，条目保留但不评分 |
